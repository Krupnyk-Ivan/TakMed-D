## 1. Огляд проєкту

**Назва:** TacMed
**Тип:** Мобільний навчальний застосунок (microlearning + gamification + AI assistant)
**Цільова платформа:** Android 8.0+ (API 26+), архітектура допускає розширення на iOS 13.0+ та Web

## 3. Архітектура

### Clean Architecture (обов’язково)

Кожен feature-модуль ділиться на **три шари**:

```
feature_name/
├── data/           # DataSources (remote/local), Models, RepositoryImpl
├── domain/         # Entities, Repository interfaces, UseCases
└── presentation/   # BLoC (events/states), Pages, Widgets
```

**Правила:**

- `domain` не знає нічого про `data` чи `presentation`
- `data` реалізує інтерфейси з `domain`
- `presentation` залежить тільки від `domain` (через UseCases)
- Усі use cases повертають `Either<Failure, T>` (Dartz)

### Offline-first стратегія

- Усі читання/записи спочатку йдуть у локальну БД Drift
- Синхронізація з Supabase — у фоні через `DataSyncService` + `workmanager`
- Механізм dirty-tracking: поля `is_dirty` (default 1) та `synced_at` (nullable) у таблицях з користувацькими даними

### Порядок ініціалізації DI (критично)

1. Інфраструктура: Supabase, AppDatabase, SharedPreferences, FlutterSecureStorage, Dio
2. Feature-модулі: Auth → Onboarding → Gamification → Learning → Quiz → MARCH → AI Chat → Profile

**Призначення:** реєстрація, вхід, відновлення пароля, автоматичне відновлення сесії через Supabase Auth.

| ID          | Вимога                                                                                                 |
| ----------- | ------------------------------------------------------------------------------------------------------ |
| **REQ-1.1** | Реєстрація за email + password через Supabase Auth                                                     |
| **REQ-1.2** | Вхід за коректною комбінацією email/password                                                           |
| **REQ-1.3** | Підтвердження реєстрації через email-посилання                                                         |

- Реактивна валідація при зміні полів + фінальна перед відправкою

**Особливості:** підтримка режиму гостя (обмежений доступ без реєстрації).

**Призначення:** структуровані курси з теоретичними та чек-лист-уроками.

| ID          | Вимога                                                                       |
| ----------- | ---------------------------------------------------------------------------- |
| **REQ-3.1** | Каталог курсів, відфільтрованих за обраним треком                            |
| **REQ-3.2** | Курс містить уроки типів: `theory`, `checklist` (та опційно `quiz`, `video`) |
| **REQ-3.3** | Перегляд теоретичного уроку з форматованим текстом, зображеннями, схемами    |
| **REQ-3.4** | Інтерактивне проходження чек-лист-уроку з відмічанням кроків                 |
| **REQ-3.5** | Позначення уроку як завершеного локально + у хмарі                           |
| **REQ-3.6** | Кешування переглянутих уроків у Drift для офлайн-доступу                     |

**Технічні деталі:**

- `LearningRepositoryImpl` використовує реактивні потоки Drift (`Stream<List<CourseEntity>>`)
- UI підписується через `BlocBuilder` — реактивно реагує на зміни БД
- Контент уроку в полі `content_json` як serialized JSON масив блоків
- `ContentBlock` — sealed клас з 5 підтипами: `TextBlock`, `HeadingBlock`, `ImageBlock`, `WarningBlock`, `InfoBlock`
- Рендеринг через єдиний `ContentBlockRenderer` (без if/else по типах)
- При завершенні уроку: позначити як виконаний → оновити лічильник курсу → зберегти `UserProgress(is_dirty=true)` → нарахувати XP через `GamificationService`
- Офлайн-завантаження через `DownloadCourseOfflineUseCase` (позначає курс `is_downloaded=true`)

| ID          | Вимога                                                      |
| ----------- | ----------------------------------------------------------- |

### 4.5. Модуль `march` (Найвищий пріоритет)

**Призначення:** інтерактивний чек-лист протоколу MARCH для польового використання.

| ID          | Вимога                                                    |
| ----------- | --------------------------------------------------------- |
| **REQ-5.1** | Чек-лист MARCH: M, A, R, C, H                             |
| **REQ-5.2** | Позначення пункту як виконано + фіксація часу             |
| **REQ-5.3** | Швидкий доступ з головного екрану (не більше 2 переходів) |
| **REQ-5.4** | Повна офлайн-функціональність                             |
| **REQ-5.5** | Скидання стану для повторного використання                |

**Реалізація:**

- `enum MarchStep` з 5 значеннями, кожне має `letterCode`, `nameUa`, `maxTimeSeconds`:
  - `massiveHemorrhage` (M, 120 с)
  - `airway` (A, 60 с)
  - `respiration` (R, 90 с)
  - `circulation` (C, 90 с)
  - `hypothermia` (H, 120 с)
- **Механізм prerequisite:** кожен крок знає попередника і не може початись, поки попередній не `completed` або `skipped`. Це гарантує клінічно правильну послідовність.
- `MarchStepState`: `pending`, `inProgress`, `completed`, `skipped`
- `MarchBloc` відстежує стан кожного кроку + час виконання

**Призначення:** мотивація через XP, серії та досягнення.

| ID          | Вимога                                                                       |
| ----------- | ---------------------------------------------------------------------------- |
| **REQ-6.1** | Нарахування XP за уроки, тести, інші дії                                     |

**Архітектура:** 3 незалежні сервіси, що зберігають дані у `SharedPreferences`:

1. **`GamificationService`** — XP та рівні
2. **`StreakService`** — серії
3. **`AchievementService`** — 15 значків

- Плюс **`StreakReminderService`** — push через `flutter_local_notifications`
- **`GamificationBloc`** агрегує всі три сервіси, надає єдиний інтерфейс UI

**Система рівнів (`enum UserLevel`, 6 значень):**
| Рівень | Поріг XP |
|---|---|
| Рекрут | 0 |
| Рядовий | 100 |
| Єфрейтор | 300 |
| Сержант | 700 |
| Медик | 1500 |
| Спеціаліст тактмеду | 3000 |

`awardXp()` повертає `true` при підвищенні рівня → ініціює `LevelUpOverlay`.
Щоденний вхід: +5 XP (один раз на день).

**15 досягнень (приклади):** "Перший крок", "Іскра" (3 дні), "Тиждень вогню" (7), "Місяць сталі" (30), "Відмінник" (100% у тесті), "Знаю MARCH", "Майстер турнікету", "Врятоване дихання", "Польовий боєць" (офлайн), "Ветеран" (усі курси).

## 5. Нефункціональні вимоги

### Продуктивність

- Холодний запуск: ≤ 3 с
- Відкриття будь-якого екрану: ≤ 1 с
- Відповідь ШІ-помічника: ≤ 10 с

### Безпека

- Усі мережеві запити — HTTPS / TLS 1.2+
- Доступ до даних — через **PostgreSQL Row Level Security (RLS)**
- JWT — у `flutter_secure_storage`
- **API-ключі сторонніх сервісів НІКОЛИ не зберігаються на клієнті**

### Офлайн-доступність

- Повна функціональність з локальної БД Drift/SQLite без інтернету
- Фонова синхронізація при відновленні зв'язку

### UX

- Material Design 3
- Адаптація до системних налаштувань (тема, шрифт)
- Будь-яка базова функція — не більше 3 переходів

### Локалізація

- Повний український інтерфейс
- Архітектура з ресурсними файлами для додавання мов

### Супроводжуваність

- Clean Architecture + DI через GetIt
- Unit-тести бізнес-логіки: покриття **≥ 50%**

### Розширюваність

- Архітектура допускає розширення на iOS та Web без змін базової кодової бази
- LLM абстраговано через Edge Function-проксі — заміна постачальника без зміни клієнта

## 6. Схема локальної БД (Drift/SQLite)

### Таблиці

#### `courses`

| Колонка             | Тип     | Обмеження                                       |
| ------------------- | ------- | ----------------------------------------------- |
| `id`                | INTEGER | PK, AUTOINCREMENT                               |
| `remote_id`         | TEXT    | NOT NULL, UNIQUE (`idx_courses_remote_id`)      |
| `title`             | TEXT    | NOT NULL                                        |
| `track`             | TEXT    | NOT NULL, CHECK IN (`'military'`, `'civilian'`) |
| `total_lessons`     | INTEGER | NOT NULL                                        |
| `completed_lessons` | INTEGER | NOT NULL                                        |

#### `lessons`

| Колонка        | Тип     | Обмеження                                                           |
| -------------- | ------- | ------------------------------------------------------------------- |
| `id`           | INTEGER | PK                                                                  |
| `course_id`    | INTEGER | NOT NULL, FK → `courses.id` ON DELETE CASCADE                       |
| `remote_id`    | TEXT    | NOT NULL, UNIQUE (`idx_lessons_remote_id`)                          |
| `type`         | TEXT    | NOT NULL, CHECK IN (`'theory'`, `'quiz'`, `'checklist'`, `'video'`) |
| `title`        | TEXT    | NOT NULL                                                            |
| `content_json` | TEXT    | NOT NULL                                                            |
| `xp_reward`    | INTEGER | NOT NULL, DEFAULT 10                                                |

#### `user_progress`

| Колонка            | Тип     | Обмеження                                                  |
| ------------------ | ------- | ---------------------------------------------------------- |
| `id`               | INTEGER | PK                                                         |
| `user_id`          | TEXT    | NOT NULL, INDEXED                                          |
| `lesson_remote_id` | TEXT    | NOT NULL, INDEXED (логічний зв'язок з `lessons.remote_id`) |
| `score`            | INTEGER | NOT NULL, CHECK (`score BETWEEN 0 AND 100`)                |
| `is_dirty`         | INTEGER | NOT NULL, DEFAULT 1                                        |
| `synced_at`        | TEXT    | NULLABLE                                                   |
| `weak_topics`      | TEXT    | NULLABLE (JSON-масив ID тем)                               |

**Індекси:**

- `idx_user_progress_user_lesson` UNIQUE `(user_id, lesson_remote_id)` — запобігає дублям
- `idx_user_progress_user_dirty` `(user_id, is_dirty)` — ефективна вибірка для синхронізації

**Критично:** при відкритті БД виконати `PRAGMA foreign_keys = ON;` (за замовчуванням у SQLite вимкнено).

## 10. Зовнішні інтерфейси

### Supabase сервіси

1. **Supabase Auth** (GoTrue REST API) — автентифікація з JWT
2. **Supabase PostgREST** — CRUD над PostgreSQL
3. **Supabase Edge Functions** (Deno, eu-central-1) — проксі до LLM, серверна логіка
4. **Supabase Realtime** (WebSocket) — реалтайм-оновлення (стрімінг відповіді ШІ)

### Локальна синхронізація

- Кастомний шар `sync` відстежує `is_dirty` у Drift
- При появі мережі надсилає батч на сервер через `workmanager`
- Формат обміну: JSON
